# Developing with Claude Code

The project ships with a Claude Code AI workflow. The `.claude/` directory contains skill, agent, guide, and protocol configs that chain "requirements → design → development → acceptance" into a reusable pipeline.

If you just want to add a feature, follow the steps below. You don't need to understand the full architecture.

## Prerequisites

- [Claude Code CLI](https://docs.anthropic.com/en/docs/claude-code) installed and logged in
- MCP Server `context7` configured (used to query Rust/React library docs)
- Project compiles and runs (see [Quick Start](getting-started-en.md))

## Generic AI Development Config

The `.claude/` config in this project is tailored for RMQTT Things. To use the same workflow in your own Rust + React project, see:

**[timzaak/web-dev-skills](https://github.com/timzaak/web-dev-skills)** — A generic Claude Code plugin covering the full PRD-to-Demo pipeline.

Install:

```shell
git clone https://github.com/timzaak/web-dev-skills.git
cd /your-project
claude --plugin-dir /path/to/web-dev-skills
```

## Shortest Loop: Add a New Feature

Say you want to add a "firmware version management" feature called `firmware-version`.

```shell
# 1. Write PRD
/t-prd firmware-version

# 2. Check PRD quality (don't skip)
/t-prd-check firmware-version

# 3. Generate technical design
/t-design firmware-version

# 4. Check design (don't skip)
/t-design-check firmware-version

# 5. Generate task plan
/t-task firmware-version

# 6. Check task plan
/t-task-check firmware-version

# 7. Execute backend development
/t-run firmware-version --phase backend

# 8. Backend finalize (format, clippy, test, export OpenAPI)
/t-backend-finalize firmware-version

# 9. Execute frontend development
/t-run firmware-version --phase frontend

# 10. Run demo tests and accept
/t-demo-run firmware-version
/t-demo-accept firmware-version
```

Each command reads the output of the previous step. For example, `/t-design` reads the PRD generated by `/t-prd`, and `/t-run` reads the task plan from `/t-task`.

## Command Reference

### Requirements and Design

| Command | Args | Output | What it does |
|---------|------|--------|-------------|
| `/t-prd <name>` | Feature name | `docs/prd/<domain>/<name>.md` | Create or update PRD with user stories |
| `/t-prd-check <name>` | Feature name | `.ai/quality/prd-check-*.md` | Validate PRD completeness and story quality |
| `/t-design <name>` | Feature name | `.ai/design/<name>.md` | Generate API design, DB schema, implementation plan |
| `/t-design-check <name>` | Feature name | `.ai/quality/design-check-*.md` | Score design implementability |
| `/t-consistency-check <module>` | Module name or `--all` | `.ai/quality/consistency-*.md` | Compare PRD vs implementation, find gaps |

### Tasks and Execution

| Command | Args | Output | What it does |
|---------|------|--------|-------------|
| `/t-task <name>` | Feature name | `.ai/task/<name>/` | Break design into executable tasks |
| `/t-task-check <name>` | Feature name | `.ai/quality/task-check-*.md` | Validate task plan executability |
| `/t-run <name> --phase <backend\|frontend\|demo>` | Feature + phase | `.ai/task/<name>/.state.json` | Drive sub-agents to execute development |
| `/t-backend-finalize <name>` | Feature name | Updates `frontend/api.json` | Format, clippy, test, export OpenAPI |

### Testing

| Command | Args | Output | What it does |
|---------|------|--------|-------------|
| `/t-backend-test-run` | None | Console output | Run Rust tests, auto-diagnose and fix failures |
| `/t-demo-run <file>` | Test file path or role name | `.ai/quality/demo-run-*.md` | Run single E2E test, auto-fix on failure |
| `/t-demo-run-all` | Optional `continue` | `.ai/quality/demo-run-all-*.md` | Run all E2E tests |
| `/t-demo-accept <name>` | Test path, role name, or `all` | `.ai/quality/demo-accept-*.md` | Accept test coverage and runnability |

## Practical Tips

### Run phases sequentially, not all at once

`/t-run` accepts a `--phase` argument:

```shell
# Backend first, make sure it compiles
/t-run firmware-version --phase backend

# Then frontend
/t-run firmware-version --phase frontend

# Then demo
/t-run firmware-version --phase demo
```

Frontend depends on the backend's OpenAPI schema. `/t-backend-finalize` exports the latest `api.json` to `frontend/`.

### Don't skip check commands

`/t-prd-check`, `/t-design-check`, and `/t-task-check` produce specific fix suggestions when quality falls short. Skipping them lets problems accumulate and become harder to fix later.

### PRDs can be iterated

If you find gaps after running `/t-prd`, run it again. It reads the existing PRD and does incremental updates without overwriting your manual edits.

### When demo tests fail

```shell
# Run a single test with auto-diagnosis and fix
/t-demo-run firmware-version

# If auto-fix doesn't work, check the diagnosis report
# Reports are at .ai/quality/demo-run-*.md
```

Diagnosis reports tell you whether the failure is a frontend, backend, or test issue, and suggest fixes.

## File Locations

```
docs/prd/              PRD documents
.ai/design/            Technical design docs
.ai/task/              Task plans and execution state
.ai/quality/           Check and acceptance reports
frontend/api.json      Exported OpenAPI schema from backend
```

## Just Fixing a Bug

For small fixes, you don't need the full pipeline. Describe what you want to change in Claude Code directly — it follows the configs in `CLAUDE.md` and `.claude/` automatically.

For bug fixes specifically:

```shell
# If it's a backend bug
/t-backend-test-run    # Confirm existing tests pass first

# After fixing the code
/t-backend-finalize <name>  # Format, clippy, test
```
