Device Auto-Provisioning PRD Preview

Status: Draft Priority: P0 Source: docs/prd/core/device-auto-provisioning.md

Introduce a devices identity table with two independent registration paths (HMAC auto-provisioning + certificate manual registration). The devices table serves as the admission gate for HMAC connections: devices without a record are denied when the product switch is OFF.

Scope

In Scope

  • devices table as device identity registry (who, how registered, when)
  • Product-level auto-provisioning toggle (per-product, default OFF)
  • HMAC path: auto-register on first auth + deny unregistered when switch OFF
  • Certificate path: create device record when admin issues a certificate (manual)
  • Registration source tagging (auto vs manual)
  • Device list display and filtering by registration source
  • Admission control: devices table as HMAC connection gate

Out of Scope

  • Device de-registration / deletion flow
  • Device blacklist (blocking specific device_id from auto-registration)
  • Custom device naming (auto-registered devices use device_id as name)
  • Dynamic registration MQTT topic protocol
  • Batch device import / pre-registration
  • Device registration approval workflow
  • Device enable/disable lifecycle management

Device Registration States

Registered (auto)

Device identity created via HMAC auto-provisioning. Source = auto. Name defaults to device_id.

Registered (manual)

Device identity created when admin issued a certificate. Source = manual.

Not registered (denied)

No devices record. HMAC connections are denied when the product switch is OFF. Device cannot connect via HMAC.

Legacy (status only)

Has a device_status row (from before this feature) but no devices row. Device will be denied on next HMAC connection if switch is OFF.

Rules

  • Admission gate: The devices table is the HMAC admission gate. A device without a record is denied connection when the product switch is OFF.
  • Product-level control, default OFF: Auto-provisioning is a per-product setting. New products default to OFF, zero impact on existing products until explicitly enabled.
  • HMAC prerequisite, no security degradation: Devices must pass HMAC-SHA1 verification before auto-registration is considered. Invalid credentials are never registered.
  • Check devices table first: The auth webhook queries devices before checking the product switch. If a record exists, the device is allowed regardless of switch state.
  • Two independent paths: HMAC and certificate paths do not depend on each other. A device registered via one path can use the other path's connection method independently.
  • Idempotent UPSERT: The same product_id + device_id combination produces exactly one device record regardless of how many times the device reconnects.
  • Registration source distinction: Every device record carries a source field -- auto for HMAC auto-provisioned, manual for certificate-issued.
  • Minimal interference: Auto-registration does not alter existing HMAC authentication, certificate issuance, property reporting, or command delivery flows.
  • No device naming: Auto-registered devices use device_id as their display name. No custom naming UI is provided.
  • Certificate path creates devices record: When admin issues a certificate, a devices record is simultaneously created with source = manual.

Acceptance

Goal Observable Result
Toggle persistence Auto-provisioning switch is visible on product edit page; state is saved and persists across page reloads.
First connection auto-registers (switch ON) When a product has auto-provisioning enabled, a device passing HMAC auth for the first time has a devices record created with source=auto and is allowed to connect.
Unregistered device denied (switch OFF) When a product has auto-provisioning disabled, a device with valid HMAC credentials but no devices record is denied connection. The device cannot connect via HMAC.
Registered device unaffected by switch A device with an existing devices record is allowed to connect regardless of whether the product auto-provisioning switch is ON or OFF.
Certificate issuance creates devices record When admin issues a certificate for a device, a devices record is simultaneously created with source=manual. The device can subsequently connect using the certificate.
Registration source visible Device list page shows a "Registration Source" column with labels (Auto / Manual) and supports filtering by source.
Idempotent registration The same device reconnecting multiple times results in exactly one devices record -- no duplicates.
Two paths independent HMAC path and certificate path operate independently. Enabling/disabling auto-provisioning has no effect on certificate-registered devices. Certificate issuance does not affect HMAC logic.

Assumptions

示例数据，不是接口契约。

  • Device uniqueness: Assumed to be product_id + device_id combination (consistent with existing device_status UPSERT pattern). Requires technical design confirmation.
  • Default device state: Auto-registered devices are assumed to be in "active" state upon creation. Independent enable/disable lifecycle management is out of scope.
  • Frontend interaction scope: The auto-provisioning toggle appears only on the product edit page, not the product creation page, to keep the creation flow simple.
  • No blacklist: There is no mechanism to block specific device_id values from auto-registering. This is explicitly out of scope.
  • Certificate path assumption: Certificate path creates the devices record at issuance time. If the certificate issuance and devices record creation are not atomic, a partial failure could leave the certificate without a devices record -- technical design must address this.
  • Legacy device_status data: Existing device_status records from before this feature will remain. Devices with status records but no devices record will be denied HMAC connections when switch is OFF, which is the intended behavior.
