## 2. 范围界定

### 2.1 包含功能

- **产线 HTTP API（独立认证）**：供产线操作员/系统分批上报；与设备 HMAC 证书体系、Admin 认证体系隔离。
- **子组件出厂元数据上报**：按子组件 SN 上报元数据，元数据同时支持**结构化字段**与**文件附件**两种形态（任一或两者并存）。
- **设备级（整机）出厂元数据上报**：按设备 SN 上报整机级元数据（如整机序列号标签、出厂检验报告、整机固件版本），与子组件级对称、独立落地。
- **设备-子组件关联上报**：按设备 SN + 子组件 SN 上报关联关系。
- **出厂元数据存储（双表、无"组装完成"概念）**：设备级与子组件级元数据独立存储；查询/读取时按 SN **left join 当前存在部分**，无状态机、无主动通知。
- **乱序异步组装**：不同批次（子组件元数据 / 设备级元数据 / 设备-子组件关联）可任意顺序到达，平台按 SN 异步落地，查询时合并当前已有部分。
- **幂等覆盖 + 变更日志**：同 SN 重复上报以最新覆盖既有记录；每次覆盖记录一条变更日志以便追溯。设备级与子组件级变更日志共用 SN 列，路径中立化。
- **管理员查询 API**：查询设备的子组件清单、出厂元数据（设备级与子组件级、结构化字段与文件附件）、变更日志。
- **设备运行时读取**：设备上线后读取自身出厂元数据（含设备级与子组件级）；部分到达时返回当前存在部分。

### 2.2 不包含功能 (Out of Scope)

- **"组装完成"判定**：不引入组装状态字段、不判定何时算组装完毕、不做孤儿 SN 告警、不做主动通知。
- **子设备网络拓扑管理**：不维护子组件之间的网络/连接拓扑关系。
- **子设备独立 MQTT 连接 / 独立认证**：子组件复用主设备连接与认证，不作为独立 MQTT 客户端。
- **数据归属与转发**：不做网关替子设备转发数据时的归属判定。
- **作为运行时 desired**：出厂元数据是"出厂一次性配置"语义，不作为运行时 desired、不复用 `property_desired`（语义不同、独立模型）。
- **复用 PropertyCommand 推送通道承载出厂元数据**：避免污染 property 语义。
- **完整版本/时间线历史**：仅保留变更日志（便于追溯），不做多版本快照、不做版本号并发控制、不做时间线 diff 查询。
- **标定数据的可视化编辑器**：管理员后台只查询，不在后台编辑出厂元数据（写入仅产线 API）。
- **子组件元数据管理后台 CRUD**：后台不提供子组件元数据的增删改入口，仅查询。

### 2.3 依赖项

- **PostgreSQL**：出厂元数据存储（设备级 + 子组件级 + 关联 + 变更日志）。
- **文件上传能力（`integration/file-upload`）**：当元数据含文件附件形态时，复用现有文件上传能力承载文件本体；元数据表只存文件引用。
- **RMQTT / MQTT Broker**：设备运行时读取依赖设备与平台的交互通道。
- **产线认证机制**：独立产线 API 认证机制，与 Herald/Admin 会话认证、HMAC 设备证书认证隔离（具体方案属设计范畴）。

## 3. 需求概述

### 3.1 功能描述

产线在生产过程中分批主动上报设备的出厂元数据：包括子组件（如摄像头、传感器等）的元数据（含标定等参数，可为结构化字段或文件附件）、设备级（整机）元数据，以及设备 SN 与各子组件 SN 的关联关系。这些批次**到达顺序不确定**，平台需把它们按 SN 异步落地，在查询/读取时合并为设备的完整视图。

出厂元数据为产线、管理员、设备三方提供闭环：

- 产线经独立 HTTP API 分批上报（独立认证），不污染设备与 Admin 认证体系。
- 平台双表独立存储、按 SN 异步 join，**无"组装完成"概念、无主动通知**；查询/读取时返回当前存在部分。
- 管理员在后台查询任意设备的子组件清单、设备级与子组件级出厂元数据与变更日志。
- 设备上线后读取自身出厂元数据生效。

核心产品语义：

- **出厂元数据 ≠ 运行时 desired**：出厂元数据是"出厂一次性配置"（可被运行期重新上报覆盖），与运行时期望状态语义不同，独立模型、互不引用。
- **无组装完成**：双表独立存储，查询时 left join 当前存在部分；无状态机、无孤儿告警、无主动通知。
- **写入收口**：出厂元数据仅由产线 API 写入；管理员后台只查询不写入。

### 3.2 关键特性

- 子组件元数据按 SN 独立落地，**与设备-子组件关联异步到达、乱序不阻塞**。
- 元数据**同时支持结构化字段与文件附件**两种形态（任一或两者并存），文件附件复用 `integration/file-upload`，元数据表只存引用。
- 设备级（整机）元数据与子组件级元数据**对称**：同样的上报/覆盖/变更日志语义，设备级无"子组件类型"概念。
- 同 SN 重复上报**幂等覆盖**，并记录**变更日志**便于追溯（非完整版本历史）。
- 部分数据未到达时，查询/读取返回当前存在部分，不报错不阻塞。

## 4. 业务规则与状态

### 4.1 业务规则

- **R1 独立认证**：产线 API 使用独立认证机制，不复用设备 HMAC 证书体系，也不复用 Admin 会话认证。具体认证形态属设计范畴。
- **R2 双表独立存储、无组装完成**：设备级元数据与子组件级元数据各自独立落地；不维护"组装完成"状态、不判定孤儿 SN、不主动通知。查询/读取时按 SN left join 当前存在部分。
- **R3 乱序异步组装**：子组件元数据、设备级元数据与设备-子组件关联可任意顺序到达；先到者先落地，后到者补充，查询结果反映当前已到达部分。
- **R4 写入收口（产线 API 唯一写入入口）**：出厂元数据仅由产线 API 写入；管理员后台接口、设备读取接口均为只读，不写入元数据。
- **R5 幂等覆盖 + 变更日志**：同一 SN 重复上报时以最新上报覆盖既有记录；每次覆盖记录一条变更日志（含覆盖前后可追溯信息），便于追溯。**不**做完整多版本快照、不做版本号并发控制、不做时间线 diff。设备级与子组件级变更日志共用 SN 列。
- **R6 结构化字段与文件附件并存**：元数据可同时包含结构化字段和文件附件；文件附件本体经 `integration/file-upload` 承载，元数据表只存文件引用。
- **R7 设备读取只读**：设备读取自身出厂元数据为只读操作；部分到达时平台返回当前存在部分，设备自行容忍缺失或稍后重读。
- **R8 语义独立**：出厂元数据与运行时 desired（`docs/prd/core/shadow-device-support.md`）为两套独立模型，互不引用、互不污染。

### 4.2 关键状态与异常

- **部分数据查询/读取**：因无"组装完成"概念，无论管理员查询还是设备读取，平台一律返回当前已到达部分；未到达部分不报错、不阻塞，不区分"未上报"与"已上报但缺失"。
- **变更追溯**：覆盖发生时记录变更日志；管理员可查询某 SN 的变更日志以追溯历史覆盖。
- **重复上报幂等**：同 SN 重复上报最终一致（覆盖语义）；并发重复上报的最终值由覆盖时序决定，平台不保证严格 last-write-wins（无版本并发控制）。
- **权限可见性**：
  - 产线 API：独立认证，仅授权的产线身份可写入。
  - 管理员查询接口：遵循现有管理端接口的访问控制原则（Herald 配置时受认证保护，未配置时单租户部署不做额外鉴权）。
  - 设备读取：设备只能读取自身 SN 命名空间下的出厂元数据，不越权访问其他设备。

## 5. 功能需求

### 5.1 核心需求

1. 产线操作员/系统可通过独立认证的产线 HTTP API 分批上报子组件出厂元数据（结构化字段和/或文件附件）。
2. 产线操作员/系统可通过同一产线 API 上报设备级（整机）出厂元数据，与子组件级对称。
3. 产线操作员/系统可通过同一产线 API 上报设备 SN 与子组件 SN 的关联关系，与元数据异步到达、乱序不阻塞。
4. 同 SN 重复上报以最新覆盖既有记录，并记录一条变更日志以便追溯。
5. 管理员可在后台查询任意设备的子组件清单、设备级与子组件级出厂元数据（结构化字段与文件附件）以及变更日志；部分数据未到达时返回当前存在部分，不报错不阻塞。
6. 设备上线后可读取自身出厂元数据（含设备级与子组件级）；部分到达时返回当前存在部分，设备自行容忍缺失或稍后重读。
7. 产线 API 与设备 HMAC 证书体系、Admin 会话认证隔离（独立认证机制）。
8. 出厂元数据与运行时 desired 互不引用、互不污染（两套独立模型）。

### 5.2 验收目标

- 产线先上报子组件元数据、后上报设备-子组件关联（或反之）时，两条记录均被平台按各自 SN 独立接收落地；管理员后续查询能看到组装后的设备视图。
- 元数据同时含结构化字段与文件附件时，两者均被记录且可被管理员与设备查询/读取。
- 同一 SN 重复上报后，平台以最新覆盖既有记录，且变更日志可追溯该次覆盖。
- 管理员查询某设备、设备读取自身元数据时，若部分子组件未到达，平台返回当前存在部分，不报错也不阻塞。
- 设备级（整机）元数据的上报、覆盖、变更日志与查询读出，与子组件级行为对称。
- 产线 API 使用与设备认证、Admin 认证均隔离的独立认证，未授权身份无法写入。
- 出厂元数据不作为运行时 desired、不被命令通道临时值污染。

## 6. API 相关约束

**适用性**: 适用

### 接口能力范围

- **产线侧能力（写入唯一入口，独立认证）**：子组件出厂元数据上报（结构化字段 + 文件附件）、设备级出厂元数据上报、设备-子组件关联上报。
- **管理端能力（只读）**：查询设备子组件清单、查询出厂元数据（设备级与子组件级、含文件附件引用）、查询变更日志。
- **设备侧能力（只读）**：设备运行时读取自身出厂元数据。
- **复用既有能力（不改语义）**：文件上传（`integration/file-upload`，承载文件附件本体）。

### 访问控制原则

- 产线 API 独立认证，与设备 HMAC、Admin 会话认证隔离；具体认证形态属设计范畴。
- 管理端查询接口遵循现有管理端接口的访问控制原则（Herald 配置时受认证保护，未配置时单租户部署不做额外鉴权）。
- 出厂元数据写入收口到产线 API；管理端与设备读取均为只读，不写入。

### 数据边界

- 出厂元数据按 SN 组织：设备级元数据按设备 SN、子组件级元数据按子组件 SN、关联按（设备 SN, 子组件 SN）。
- 设备读取仅限自身 SN 命名空间下的元数据，不越权访问其他设备。
- 文件附件本体存储边界遵循 `integration/file-upload` 现有约定；元数据表只存引用。

### 兼容性要求

- 不修改既有 `property_command`、`property_desired`、reported 快照、属性回调的状态机与语义。
- 不修改既有文件上传能力的语义；出厂元数据复用其承载文件附件。

> 具体端点、请求/响应参数表、HTTP 状态码、表结构属设计范畴，由技术设计承接，不在本 PRD 展开。

## 7. 前端/交互约束

**适用性**: 适用（仅查询展示，无写入入口）

### 页面入口

- 在设备管理后台的设备上下文中提供"出厂元数据/子组件清单"区域（与现有设备详情/设备管理页保持一致的导航层级）。

### 关键交互

- 展示该设备的设备级元数据、子组件清单，每个子组件的元数据（结构化字段与文件附件引用），以及可下钻查看变更日志。
- 文件附件以可识别的引用形式展示（如可下载链接），不在后台编辑。
- 部分子组件未到达时，已到达部分正常展示，未到达部分不报错（与"返回当前存在部分"语义一致）。

### 状态反馈

- 查询成功/失败的即时反馈。
- 变更日志按时间倒序展示，支持追溯同一 SN 的覆盖历史。

### 权限可见性

- 前端出厂元数据查询与现有设备管理界面遵循同一访问控制；产线上报不在后台前端暴露（产线 API 独立认证）。

## 8. 已确认决策

- **真问题定义**：产线主动上报的、乱序到达的、含设备-子组件关联的出厂元数据，需平台异步组装，供后台查询 + 设备运行时读取。
- **元数据语义**：出厂元数据 ≠ 运行时 desired，独立模型（不复用 `property_desired`）。
- **上报认证**：独立产线 HTTP API，不复用设备 HMAC 证书体系，也不复用 Admin 会话认证。
- **组装时序**：乱序到达，异步组装。
- **不引入"组装完成"概念**：无主动通知、无状态机、无孤儿告警；双表独立存储，查询时 left join 当前存在部分。
- **设备读取约定**：部分到达时平台返回当前存在部分，设备自行容忍缺失或稍后重读。出厂元数据语义与 property 不同，不复用 PropertyCommand 通道，避免污染 property 语义。
- **子组件元数据形态**：元数据同时支持结构化字段与文件附件（任一或两者并存）；文件附件复用 `integration/file-upload`。描述泛化，不限定单一组件类型（摄像头仅为示例）。
- **运行期变更**：出厂元数据运行期可被重新上报覆盖；保留**变更日志**便于追溯（非完整版本/时间线历史、无版本号并发控制）。
- **写入收口**：出厂元数据仅由产线 API 写入；管理员后台与设备读取均为只读。
- **设备级对称扩展**：设备级（整机）元数据复用子组件级的上报/覆盖/变更日志语义，独立落地，变更日志共用 SN 列。
