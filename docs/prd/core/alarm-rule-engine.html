Scope

In Scope

  • Rule management: create / list / edit / enable-disable / delete
  • Four trigger types: property threshold, event match, device online, device offline
  • Single condition with operators: >, >=, <, <=, ==, !=, between, contains, always
  • Actions: create alarm record (mandatory), Webhook callback (optional)
  • Alarm deduplication: rule + device dimension, throttle_minutes window
  • Alarm record management: list with filters, acknowledge
  • Rule scoped per product_id
  • Async rule evaluation, non-blocking data ingestion path

Out of Scope

  • Multi-condition compound logic (AND/OR nesting)
  • Per-device rule scoping (product-level only)
  • Notification push (email / SMS / DingTalk / WeCom) -- deferred to notification system
  • Alarm statistics / aggregation dashboard
  • Alarm escalation and workflow
  • Rule versioning and audit log
  • MQTT command action (V2 consideration)

Enable/Disable is a toggle. Deleted is terminal (removed from list). Rules in Disabled state are skipped during evaluation.

Alarm Record Lifecycle

Triggered (Unconfirmed)
→
Confirmed (Acknowledged)

Confirmation is one-way. Confirmed alarms cannot be un-confirmed. The acknowledge button is hidden for already-confirmed records.

Trigger Type States

Property Threshold

Triggered when a device reports a property value meeting the configured operator + threshold. E.g. temperature > 50.

Event Match

Triggered when a device reports an event matching the configured event identifier. Condition uses "contains" or "always".

Device Online

Triggered when a device under the product connects. No additional condition needed (always).

Device Offline

Triggered when a device under the product disconnects. No additional condition needed (always).

Rules

  • Product-scoped: Each rule is bound to exactly one product_id. Rules are evaluated only for devices belonging to that product.
  • Single condition per rule: V1 supports one trigger type + one condition per rule. No AND/OR compound logic.
  • Async evaluation: Rule evaluation runs after data storage completes. It does not affect property/event/connect/disconnect response latency.
  • Deduplication (throttle_minutes): When a rule triggers for a specific device, subsequent triggers for the same rule + same device are suppressed within the configured throttle window. The key is (rule_id, device_id).
  • Disabled rules are invisible to evaluator: When a rule is disabled, it is excluded from the cached rule set. Re-enabling restores it to active evaluation.
  • Mandatory action -- create alarm record: Every triggered rule must create an alarm record containing the alarm level, trigger data, and message.
  • Optional action -- Webhook callback: If configured, an HTTP POST is sent to the configured URL. Failure does not block alarm record creation.
  • Alarm levels: info, warning, critical. Set at rule creation time, stored in each alarm record.
  • Confirmation is idempotent: An already-confirmed alarm cannot be re-confirmed. The acknowledge action is not shown for confirmed records.
  • Product field immutable after creation: Once a rule is created, its product_id cannot be changed during edit.
  • Webhook failure strategy (tentative): Current assumption is to record failure status in the alarm record without automatic retry. Retry strategy to be confirmed during design phase.

Assumptions

以上线框图使用了示例数据。示例数据，不是接口契约。

  • Webhook failure strategy (pending confirmation): The PRD states "record failure status to alarm record, no automatic retry." This affects the action executor complexity. Whether retry is needed should be confirmed during design phase based on Platform Admin usage feedback.
  • Rule cache invalidation: Assumed to follow the existing metadata cache pattern (loaded on startup, invalidated on rule CRUD). Cache strategy details deferred to technical design.
  • Alarm record retention: No retention policy or automatic cleanup is specified in the PRD. Assumed that records persist indefinitely until manually managed.
  • Concurrent trigger handling: Assumed that the async evaluation handles concurrent triggers for different devices of the same product independently without race conditions on the dedup cache.

Confirmed Decisions (from PRD)

  • Notification push (email/SMS/DingTalk/WeCom) is NOT in V1 scope -- deferred to notification system.
  • V1 only supports single condition per rule -- no AND/OR compound logic.
  • Alarm acknowledgement IS in V1 scope.
  • Rules are scoped per product only -- no per-device rules.
  • Alarm statistics/aggregation dashboard is NOT in V1 scope.
  • MQTT command action deferred to V2.
