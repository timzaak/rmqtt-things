## 1. 产品管理

### 故事 1：创建产品 [US-PA-001]

**优先级**: P0

**【用户故事】**
**作为**：Platform Admin
**我希望**：在后台创建新的 IoT 产品，填写名称、型号和描述
**从而**：将同类设备归入同一产品进行统一管理

**【验收标准】**

**场景 1：正常创建产品**
```gherkin
Given 管理员已登录后台
When 在产品列表页点击"Create Product"，填写产品名称、型号编号并提交
Then 系统创建产品，页面跳转到产品列表，新产品出现在列表中
```

**场景 2：型号编号重复**
```gherkin
Given 系统中已存在型号编号为 "SENSOR-A01" 的产品
When 管理员创建新产品时使用相同型号编号
Then 系统提示创建失败，产品未被创建
```

## 2. 证书管理

### 故事 4：签发设备证书 [US-PA-004]

**优先级**: P0

**【用户故事】**
**作为**：Platform Admin
**我希望**：为指定产品的设备签发 TLS 证书，设定有效期
**从而**：设备可使用证书进行 MQTT 安全连接

**【验收标准】**

**场景 1：正常签发证书**
```gherkin
Given 管理员已登录后台，且存在目标产品
When 管理员选择产品、输入设备 ID、设定起止时间并提交
Then 系统签发证书，签发成功后页面展示证书内容和私钥内容，并提供下载按钮
```

**场景 2：设备已有有效证书**
```gherkin
Given 指定设备已存在状态为 Normal 且未过期的证书
When 管理员尝试为该设备再次签发证书（未勾选强制重签）
Then 系统提示"证书已存在且有效"，拒绝签发
```

**场景 3：强制重签证书**
```gherkin
Given 指定设备已存在有效证书
When 管理员勾选"Force re-issue"并提交
Then 系统重新签发证书，新证书替代旧证书
```

## 4. OTA 固件管理

### 故事 11：创建 OTA 版本 [US-PA-011]

**优先级**: P1

**【用户故事】**
**作为**：Platform Admin
**我希望**：在创建页面填写版本信息并上传固件文件，完成 OTA 版本创建
**从而**：准备向设备推送固件升级

**【验收标准】**

**场景 1：正常创建 OTA 版本**
```gherkin
Given 管理员已登录后台
When 管理员在创建页面选择产品、填写版本 key、版本号（x.y.z 格式）、最小版本、最大版本号（可选）、上传固件文件后提交
Then 系统创建 OTA 版本记录，页面跳转到 OTA 版本列表
```

**场景 2：上传固件文件**
```gherkin
Given 管理员正在创建 OTA 版本
When 管理员选择固件文件并触发上传
Then 系统自动计算文件大小和 MD5 哈希值，将文件上传至对象存储，并在表单中展示 file_key、bin_length 和 bin_md5
```

**场景 3：版本格式校验**
```gherkin
Given 管理员正在填写版本号字段
When 输入的版本号不符合 x.y.z 格式（如 "abc" 或 "1.2"）
Then 页面提示格式错误
```

**场景 4：指定设备范围**
```gherkin
Given 管理员正在创建 OTA 版本
When 管理员逐个添加目标设备 ID 并提交
Then 该固件版本仅对列表中的设备生效
```

**场景 5：必填字段缺失**
```gherkin
Given 管理员正在创建 OTA 版本
When 未填写产品、版本 key、版本号、最小版本、最大版本号或未上传固件文件就提交
Then 页面提示缺少必填字段，不发送创建请求
```

**场景 6：未保存离开提示**
```gherkin
Given 管理员已在创建页面填写了部分信息
When 管理员尝试离开当前页面
Then 页面弹出确认提示，防止意外丢失填写内容
```

**场景 4：不设置持续时间时保持即时触发**

```gherkin
Given 管理员正在创建告警规则

```

**场景 5：持续时间输入无效值**


**场景 2：离线设备设置期望状态后排队投递**

**场景 5：用 null 删除单个期望属性**
```gherkin

When 管理员提交 patch，其中 colorTemp 为 null
```

### 故事 45：分批上报子组件元数据 [US-PA-045]

**优先级**: P0

**【用户故事】**
**作为**：产线操作员/产线系统（详见 [docs/user-stories/_roles.md](/_roles.md)）
**我希望**：通过独立产线 API 分批上报设备的子组件元数据（含结构化字段和/或文件附件），即便不同批次到达顺序不同
**从而**：每个子组件（如摄像头、传感器）的出厂元数据（含标定等参数）都能被平台独立接收并按 SN 落地，不依赖上报顺序

**【验收标准】**

**场景 1：先上报子组件、后上报设备-子组件关联（乱序正常落地）**
```gherkin
Given 产线尚未上报某设备 SN 的任何信息
When 产线先上报一个子组件 SN 的元数据，稍后再上报设备 SN 与该子组件 SN 的关联
Then 两条记录都被平台独立接收并按各自 SN 落地
And 管理员后续查询该设备时能看到该子组件及其元数据
```

**场景 2：结构化字段与文件附件混合上报**
```gherkin
Given 一个子组件的元数据既有结构化参数（如标定数值）也可能有文件附件
When 产线上报该子组件的元数据
Then 结构化字段与文件附件都被平台记录并可被查询
```

**场景 3：同 SN 重复上报（幂等覆盖）**
```gherkin
Given 某子组件 SN 的元数据已被上报过
When 产线再次上报同一子组件 SN 的元数据
Then 平台以最新上报覆盖既有记录
And 该次覆盖被记录为一条变更日志以便追溯
```
